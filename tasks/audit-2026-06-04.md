## P1 — fix soon

- **P1 atelier-cli/src/main.rs — observability — the CLI binary never installs a tracing subscriber, so every `tracing::warn!`/`info!` in `runner.rs` is silently dropped (GUI and TUI both install one; `grep -rln tracing_subscriber crates/*/src` confirms cli absent) — install a default `tracing_subscriber::fmt()` writing WARN+ to stderr in `main()`.**
  Downstream casualties of this one gap: session-save failures (runner.rs:1539), the "DoD config loaded but checks not wired" warning (runner.rs:2820), probe failures, `pane_visibility.json` write failures — all invisible to a CLI user today.
- **P1 atelier-cli/src/runner.rs:2805 — correctness — max_turns exhaustion leaves `final_state = Streaming`; `exit_code_for_final_state(Streaming)` = 0, so CI/scripts treat a timed-out run as success — set `final_state = AwaitingUser` when the loop exhausts without a terminal transition.**
  The doc comment on `run()` (runner.rs:2366) already promises "`max_turns` reached (timeout; `final_state = AwaitingUser`)" — the code doesn't deliver it. Contract exists; implementation drifted. `exit_code_for_final_state` maps AwaitingUser→6, everything else→0 (lib.rs:89).
- **P1 atelier-cli/src/runner.rs:1539 — durability — `snapshot.save_split_to` failure is warn-only and the run still exits 0; combined with the missing subscriber the user gets zero indication their session wasn't persisted; next `--resume` silently reads stale state — at minimum `eprintln!` the failure; consider a nonzero exit or a `RunReport` field.**
  The best-effort rationale comment predates v61's "write the conversation back so `--resume` has something to reconstruct from" — resume now depends on this write succeeding.
- **P1 workspace deps — RUSTSEC-2026-0009 — `time` 0.3.41 DoS via stack exhaustion (transitive) — documented audit-ignore debt per v60.85 while pinned to Rust 1.85; revisit when the toolchain pin moves.**

## Summary

The workspace is in good shape: every deterministic gate is green, the security-sensitive surfaces (path containment, egress, credentials, terminal output, IPC) show deliberate defence-in-depth with audit-trail comments, and the test rig actively enforces past lessons. The real findings concentrate in one architectural seam: **the CLI driver's failure-reporting path**. A run that times out reports success (exit 0), a session that fails to persist reports success, and the warnings that would have disclosed both are emitted into a tracing void because the binary never installs a subscriber. All three compound each other and all three are cheap to fix — subscriber install, one `final_state` assignment after the loop, one `eprintln!`/report-field on persist failure. Recommend bundling them as a single "CLI failure-visibility" PR with a regression test asserting exit code 6 on max_turns exhaustion.
